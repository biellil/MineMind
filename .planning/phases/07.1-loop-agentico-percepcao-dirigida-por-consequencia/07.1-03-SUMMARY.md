# Phase 07.1 Plan 03: Driver Event-Driven com makeParkPromise Summary

**One-liner:** Driver event-driven com makeParkPromise (Promise.race sobre actionFinished/abort/timeout-piso) substituindo sleep fixo, enteredIdle/nextWakeMs no grafo, e heartbeat em timers autônomos.

## What Was Built

Esta é a mudança central da Fase 07.1 — transforma o ticker baseado em relógio em cadeia agêntica onde cada ação acordar o loop ao terminar.

### Task 1: LoopAnnotation, LoopState, NodeDeps e nó execute/observe

**src/cognition/graph.ts** — dois novos campos no `LoopAnnotation`:
- `enteredIdle: Annotation<boolean>` — sinal do grafo de que não há objetivo ativo
- `nextWakeMs: Annotation<number>` — quanto tempo o park deve aguardar no máximo

**src/cognition/nodes.ts** — três mudanças:
1. `LoopState` ganha `enteredIdle: boolean` e `nextWakeMs: number`
2. `NodeDeps` ganha `triggerBus: TriggerBus` (importação inline de tipo)
3. Nó `observe` calcula:
   ```typescript
   const enteredIdle = (holder.currentGoal == null) && (snapshot !== null)
   const nextWakeMs = enteredIdle ? config.idleWakeIntervalMs : config.navigateTimeoutMs
   ```
4. Nó `execute` emite `triggerBus.emit('actionFinished', ...)` em 4 pontos, **sempre** após `holder.lastObservedDelta` atribuído (Pitfall 1 / T-07.1-10):
   - Ramo sem skill (sem objetivo): emit com `{ skill: null, outcome: null }`
   - Ramo anti-repeat/abandon: emit com `{ skill, outcome: 'no_effect' }`
   - Ramo catch (exceção genuína): emit dentro do catch + return early
   - Ramo try sucesso/falha: emit após o try/catch com `skillOutcome` capturado

### Task 2: Driver event-driven em loop.ts

**src/cognition/loop.ts** — reescrita completa mantendo toda a lógica existente:

**Adicionado:**
```typescript
function makeParkPromise(
  triggerBus: TriggerBus,
  abortSignal: AbortSignal,
  nextWakeMs: number,
): Promise<WakeReason>
```
- `WakeReason = 'actionFinished' | 'timeout' | 'abort'`
- `cleanup()` interno remove listener e clearTimeout ao acordar (sem leak)

**`startCognitiveLoop` — mudanças:**
- `TriggerBus` instanciado por sessão com `setupMineflayerListeners` + `cleanupTriggerBus()`
- `buildGraph` agora recebe `{ bot, holder, provider, triggerBus }`
- `AbortController sessionAbort` — `bot.once('end')` chama `sessionAbort.abort()`
- `bot.once('end')` também chama `cleanupTriggerBus()` e `clearInterval(flushTimer/pruneTimer)`
- **Timers autônomos** (D-05): `setInterval` para flush (30s) e prune (60s) — independem do tick
- **Race event-driven** ao final de cada iteração:
  ```typescript
  nextWakeMs = result.nextWakeMs ?? config.navigateTimeoutMs
  const wakeReason = await makeParkPromise(triggerBus, sessionAbort.signal, nextWakeMs)
  if (wakeReason === 'abort') break
  ```
- `enteredIdle` lido do resultado real do grafo: `result.enteredIdle === true`

**Removido:**
- `const sleep = (ms: number) => ...`
- `const started = Date.now()` e `const elapsed = Date.now() - started`
- `if (elapsed < config.minTickMs) await sleep(config.minTickMs - elapsed)`
- `let lastFlushAt` e `let lastPruneAt` e seus if-blocks no while
- `enteredIdle: false` hardcoded (~linha 165 do arquivo anterior)

## Deviations from Plan

### Auto-fixed Issues

Nenhuma — plano executado exatamente como escrito, com uma decisão de implementação menor:

**[Implementação] catch faz return early após emit**

O plano descrevia a estrutura de fluxo do execute, mas não especificava explicitamente se o `catch` deveria fazer `return` antes do emit externo ou usar uma variável de controle. Optou-se por `return { memory: holder.memory }` dentro do `catch` após o emit — isso é mais claro e evita duplicação do emit (o emit externo fica no caminho normal após o try/catch, usando `skillOutcome` capturado pela variável `let`). Esta abordagem é mais segura contra double-emit do que a alternativa de `skillOutcome` nulo no catch sem return.
