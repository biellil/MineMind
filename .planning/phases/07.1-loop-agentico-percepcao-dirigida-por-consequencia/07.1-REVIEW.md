## Warnings

### WR-01: `makeParkPromise` não verifica `signal.aborted` antes de adicionar listener

**File:** `src/cognition/loop.ts:50`

**Issue:** Se `bot.end` disparar durante `graph.invoke()` (que é `await`-ado), `sessionAbort.abort()` é chamado no handler de `'end'` antes de `makeParkPromise` ser invocado na próxima iteração do `while`. Quando `makeParkPromise` é então chamado com um `AbortSignal` já abortado, `addEventListener('abort', onAbort, { once: true })` nunca dispara — por spec WHATWG, o evento `'abort'` só é emitido no momento em que `abort()` é chamado; listeners adicionados após já não recebem o evento. O resultado é que a promise só resolve via `timeout` (até `nextWakeMs` ms, potencialmente 60 segundos no estado idle), e o `wakeReason === 'abort'` nunca é visto. A sessão acaba encerrando via `while (alive)` na próxima iteração, mas desperdiça o período de timeout.

**Fix:**
```typescript
function makeParkPromise(
  triggerBus: TriggerBus,
  abortSignal: AbortSignal,
  nextWakeMs: number,
): Promise<WakeReason> {
  // Verificação antecipada: sinal já abortado antes de qualquer await
  if (abortSignal.aborted) return Promise.resolve('abort')

  return new Promise<WakeReason>((resolve) => {
    // ... resto da implementação atual
  })
}
```

### WR-02: Bug de acumulação de importância quando o buffer de memória sofre evicção

**File:** `src/cognition/loop.ts:241-244`

**Issue:** `seenEvents` rastreia `events.length` para acumular importância apenas dos eventos novos. Quando o buffer de tokens expulsa eventos antigos (evicção FIFO em `push()`), `events.length` pode cair abaixo de `seenEvents`. A condição `if (events.length > seenEvents)` falha, e todos os eventos adicionados após a evicção — que são genuinamente novos — ficam invisíveis ao acumulador até `events.length` voltar a superar o valor antigo de `seenEvents`. Exemplo: `seenEvents=8`, evicção reduz para `events.length=3`, novo push → `4`. Condição `4 > 8` é falsa. O evento novo não acumula importância. O gatilho de reflexão pode nunca disparar mesmo com atividade significativa.

**Fix:**
```typescript
const events = getEvents(holder.memory)
// Detectar shrinkage: buffer evictou eventos → reset do cursor para o fim atual
if (events.length < seenEvents) {
  seenEvents = 0  // ou events.length, para tratar todos os atuais como "novos desta sessão"
}
if (events.length > seenEvents) {
  for (const e of events.slice(seenEvents)) reflState.importanceAccum += importanceOf(e)
}
seenEvents = events.length
```

Alternativamente, usar um contador monotônico de eventos em vez de `events.length` (requer mudança no contrato de `ShortTermMemory`).

### WR-03: Documentação incorreta do campo `timeOfDay` em `BotStatus`

**File:** `src/perception/types.ts:66`

**Issue:** O comentário documenta `timeOfDay` como `"0.0 (meia-noite) a 1.0 (meia-noite seguinte); < 0.5 = dia"`, mas `bot.time.timeOfDay` retorna o valor bruto de ticks do Minecraft (0–24000), não um valor normalizado 0.0–1.0. O `snapshot.ts` (linha 36) e `trigger-bus.ts` (linha 92) usam corretamente `< 13000` como limiar de dia. O comentário errado pode levar consumidores futuros do tipo a implementar lógica errada (ex.: `timeOfDay < 0.5` nunca seria verdadeiro).

**Fix:**
```typescript
// src/perception/types.ts:66
readonly timeOfDay: number    // ticks Minecraft (0–24000); 0 = amanhecer, 6000 = meio-dia, 13000 = anoitecer
```

### WR-04: Debounce trailing-edge para `hostileNearby` pode atrasar ou suprimir o sinal durante aproximação ativa

**File:** `src/cognition/trigger-bus.ts:111-115`

**Issue:** O debounce reinicia (`clearTimeout` + `setTimeout`) a cada `entityMoved`. Se um mob hostil se aproxima continuamente enquanto o Mineflayer dispara `entityMoved` a ~20Hz, o timer é reiniciado indefinidamente e `hostileNearby` só dispara quando o mob para de se mover (fim do burst). Em um cenário de creeper correndo direto para o bot, o sinal pode nunca chegar antes do impacto. O comentário documenta isso como intencional ("só o último evento no burst conta"), mas o efeito prático é que a preempção de skill depende de um sinal que pode chegar tarde demais.

**Fix para fase atual:** Implementar debounce leading-edge (dispara no primeiro evento do burst e ignora os subsequentes por `hostileDebounceMs`):
```typescript
const checkHostile = () => {
  if (this._hostileDebounce) return  // burst ativo — líder já agendado, ignorar
  const botPos = bot.entity?.position
  if (!botPos) return
  const hasHostile = Object.values(bot.entities).some(
    (e) =>
      (e as unknown as Record<string, string>).kind === 'Hostile mobs' &&
      e.position.distanceTo(botPos) < cfg.hostileRadius,
  )
  if (hasHostile) {
    this.emit('hostileNearby')
    // Bloqueia re-emissão pelo período de cooldown
    this._hostileDebounce = setTimeout(
      () => { this._hostileDebounce = undefined },
      cfg.hostileDebounceMs,
    )
  }
}
```

### IN-04: `skillRegistry[skill!]!` usa asserção de não-nulo desnecessariamente frágil

**File:** `src/cognition/nodes.ts:272`

**Issue:** A expressão `skillRegistry[skill!]!(bot, params)` usa dois operadores `!`. O primeiro (`skill!`) é seguro pois o branch `if (!skill)` já retornou antes. O segundo (`skillRegistry[skill!]!`) suprime o TypeScript de verificar que o skill existe no registry. Se `skill` for atribuído a um valor não registrado (ex.: durante extensão futura do código), a expressão lançará `TypeError: undefined is not a function` em runtime sem warning em tempo de compilação.

**Fix:**
```typescript
const skillFn = skillRegistry[skill]
if (!skillFn) {
  log(`skill desconhecida: ${skill} — tick degradado para idle`)
  triggerBus.emit('actionFinished', { skill, outcome: 'error' })
  return { memory: holder.memory }
}
const result = await skillFn(bot, params)
```
