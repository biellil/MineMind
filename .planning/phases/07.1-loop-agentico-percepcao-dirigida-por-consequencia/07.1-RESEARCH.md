### Locked Decisions

**Modelo do loop / aresta de retorno:**
- D-01: A "aresta de retorno" permanece FORA do grafo. O driver externo `while` é mantido, mas a pausa troca de `sleep(minTickMs)` por espera-por-evento: `Promise.race([actionFinished, bot.end, timeout-piso])`.
- D-02: O StateGraph segue FINITO-por-tick (observe→analyze→updateMemory→decide→execute→END). NÃO migrar para ciclo/interrupt.
- D-03: Preserva decisão do Roadmap v2.0 — "System 1 = função pura no driver, FORA do StateGraph". Opção B (ciclo no grafo + interrupt) rejeitada.
- D-04: Preservar invariantes: lock single-flight do LLM (`inFlight`) continua no deliberator; stop-on-disconnect continua trivial; flush periódico; holder como fonte única.
- D-05: Heartbeat (deadTicks, flush periódico, poda do checkpointer) passa a ser conduzido por timers próprios no driver.

**Roteador código-vs-LLM:**
- D-06: Roteador = arbiter como classificador + outcome como gatilho. `nextMechanicalStep(snapshot, holder, safety): Step | null` — devolve passo mecânico determinístico ou `null` (sinal de escalar ao LLM). O arbiter precisa de caminho `null` distinto do fallback atual.

- D-09: Park no driver via `Promise.race([evento-do-TriggerBus, timer-auto-agendado])`. O agente agenda o próprio wake (`setTimeout`).

**Gatilhos / TriggerBus:**

- D-13: Anti-tick-disfarçado: eventos de alta frequência NUNCA assinados crus — só via edge-detection com limiar/histerese/debounce.

## Summary

Esta fase substitui o driver de tick fixo (`while` + `sleep(minTickMs)`) por uma cadeia agêntica event-driven onde o término de cada ação (`actionFinished`) dispara imediatamente o próximo ciclo de percepção-decisão. A mudança é cirúrgica: a estrutura do grafo (StateGraph finito-por-tick) é preservada integralmente, e a "aresta de retorno" continua sendo o driver externo — mas a pausa entre iterações deixa de ser relógio e passa a ser `Promise.race` sobre eventos semânticos do TriggerBus.

Quatro mecanismos são entregues em conjunto: (1) a substituição do `sleep` pelo race event-driven no driver; (2) o arbiter promovido a classificador de passos mecânicos com caminho `null` (despacho ao LLM); (3) o TriggerBus como EventEmitter dedicado com edge-detection sobre eventos nativos do Mineflayer; (4) AbortController/AbortSignal como contrato de preempção entre o driver e as skills.

A leitura do codebase revelou dois achados não-óbvios com impacto direto no plano: (a) `entity.kind` já existe no Mineflayer como campo nativo populado via `minecraft-data` com o valor `'Hostile mobs'` para mobs hostis — o pré-requisito `EntityInfo.kind` é uma adição de um campo ao tipo TypeScript e ao snapshot, não uma classificação custom; (b) `bot.pathfinder.stop()` simplesmente seta `stopPathing = true`, o que faz o `goto` rejeitar com `PathStopped` — o AbortController apenas precisa chamar esse método no handler de `abort`.

**Primary recommendation:** Implementar em 4 planos sequenciais: TriggerBus+edge-detection (Plan 1), driver event-driven+park/wake (Plan 2), roteador mecânico+arbiter-classifier (Plan 3), preempção AbortController+nó execute (Plan 4). A ordem importa: TriggerBus precisa existir antes de o driver poder fazer o race sobre ele.

## Existing Code Inventory (o que existe hoje)

### loop.ts — o arquivo central da reescrita

O driver atual (linhas 106–221) é um `void (async () => { while (alive) { ... await sleep(...) } })()`. Os componentes que PERMANECEM após a reescrita:
- `bot.on('chat', ...)` handler — mantido intacto (D-04)
- `bot.once('end', ...)` + `alive = false` — mantido; agora também resolve a Promise do park
- `bot.on('death')` / `bot.on('respawn')` — mantidos como logs
- `persistHolder` no `end` e periódico — mantidos; apenas o timer passa a ser autônomo (D-05)
- `checkpointer.deleteThread` periódico — mantido; timer autônomo (D-05)
- `deliberator.maybeDeliberate` + `pickTrigger` — mantidos; o roteador COMPLEMENTA (D-08)
- `shouldReflect` / `reflState` — mantido; o `enteredIdle: false` hardcoded passa a ser preenchido pelo sinal do grafo (D-10)

O que MUDA:
- `await sleep(config.minTickMs - elapsed)` → `await wakeRace(triggerBus)` (Promise.race event-driven)
- `deadTicks` continua mas como timer autônomo, não contador de ticks (D-05)

### graph.ts — sem mudança estrutural

O `LoopAnnotation` ganha dois campos novos: `enteredIdle: boolean` e `nextWakeMs: number`. O resto do grafo é preservado integralmente (D-02).

### nodes.ts — mudanças em `execute` e `observe`

O nó `execute` (~linha 238): recebe AbortController injetado pelo driver, cria listener do TriggerBus para disparar abort, captura o resultado mesmo em abort via `finally`/catch existente (D-18).

O nó `observe`: retorna `enteredIdle` + `nextWakeMs` no estado (D-10). `enteredIdle = true` quando `cogState` transiciona para `'idle'` e `currentGoal == null`.

### arbiter.ts — promoção a classificador

A função `arbitrate` atual retorna `CognitiveState` com fallback `'exploring'` — nunca retorna `null`. Para D-06, o novo `nextMechanicalStep` retorna `Step | null` onde `null` sinaliza "decisão real → LLM". O `arbitrate` existente continua como está; o classificador é uma nova função que o chama e adiciona o caminho `null`.

### executor.ts — adição de AbortSignal

`ExecuteOptions` ganha campo opcional `signal?: AbortSignal`. No `executeWithSafety`, o `signal` é adicionado ao `Promise.race` como um quarto racer: `signal.aborted ? reject(new AbortError()) : signal.addEventListener('abort', () => reject(new AbortError()))`. [VERIFIED: codebase grep]

### perception/types.ts — adição de `kind`

`EntityInfo` precisa de `readonly kind: string` (valores: `'Hostile mobs'`, `'Passive mobs'`, `'Vehicles'`, `'Immobile'`, `'Projectiles'`, `'UNKNOWN'`). [VERIFIED: minecraft-data categories confirmadas em runtime]

### snapshot.ts — populando `kind`

No `map` de entidades (~linha 46), adicionar:
```typescript
kind: (e as unknown as Record<string, string>).kind ?? 'UNKNOWN',
```
O Mineflayer já popula `entity.kind` via `setEntityData` → `entity.kind = entityData.category`. [VERIFIED: leitura de node_modules/mineflayer/lib/plugins/entities.js linha 167]

### skills/navigate.ts e dig.ts — honrar AbortSignal

`navigate.ts`: no `executeWithSafety`, passa `signal`. No handler de abort: `bot.pathfinder.stop()`.

`dig.ts`: no `executeWithSafety`, passa `signal`. No handler de abort: `bot.pathfinder.stop()` (para o collectBlock que usa pathfinder internamente) e `bot.dig` não tem cancelamento nativo — o timeout/watchdog existente já resolve; o abort apenas encurta o `executeWithSafety`.

`bot.pathfinder.stop()` funciona porque o `goto` interno assina `'path_stop'` e rejeita com `PathStopped` quando `stopPathing = true`. [VERIFIED: leitura de node_modules/mineflayer-pathfinder/index.js linha 162 e goto.js linha 39]

## Architecture Patterns

### Padrão 1: Promise.race para park/wake event-driven

```typescript
// [ASSUMED] — padrão canônico Node.js; verificado via codebase grep no executor.ts existente
function makeParkPromise(
  triggerBus: EventEmitter,
  signal: AbortSignal,
  nextWakeMs: number,
): Promise<WakeReason> {
  return new Promise<WakeReason>((resolve) => {
    let timer: ReturnType<typeof setTimeout> | undefined

    const onActionFinished = () => { cleanup(); resolve('actionFinished') }
    const onAbort = () => { cleanup(); resolve('abort') }

    timer = setTimeout(() => { cleanup(); resolve('timeout') }, nextWakeMs)
    triggerBus.once('actionFinished', onActionFinished)
    signal.addEventListener('abort', onAbort, { once: true })

    function cleanup() {
      clearTimeout(timer)
      triggerBus.off('actionFinished', onActionFinished)
    }
  })
}
```

O `signal` aqui é o sinal de `bot.end`/morte — quando o bot desconecta, o AbortController do driver é abortado e a Promise resolve imediatamente sem deadlock (D-11).

### Padrão 2: TriggerBus como EventEmitter dedicado com edge-detection

```typescript
// [ASSUMED] — padrão Node.js EventEmitter; estrutura interna a critério do Claude (Discretion)
import { EventEmitter } from 'node:events'

export type TriggerEvent = 'actionFinished' | 'nightFell' | 'dayBroke' | 'hostileNearby' | 'stuck' | 'hungry'

export class TriggerBus extends EventEmitter {
  // Edge-state para derivação
  private _wasDay = true
  private _hostileDebounceTimer: ReturnType<typeof setTimeout> | undefined
  private _lastFoodValue = 20

  // Assinado em startCognitiveLoop, desassinado no cleanup
  setupMineflayerListeners(bot: Bot, config: TriggerConfig): () => void {
    const onHealth = () => {
      // hungry: limiar + histerese
      if (bot.food <= config.hungryThreshold && this._lastFoodValue > config.hungryThreshold) {
        this.emit('hungry', { food: bot.food })
      }
      this._lastFoodValue = bot.food
    }

    const onTime = () => {
      const isDay = bot.time.timeOfDay < 13000
      if (!isDay && this._wasDay) this.emit('nightFell')
      if (isDay && !this._wasDay) this.emit('dayBroke')
      this._wasDay = isDay
    }

    const onEntitySpawnOrMoved = () => {
      // hostileNearby: filtro por kind + distância + debounce
      const hasHostile = Object.values(bot.entities).some(
        (e) => e.kind === 'Hostile mobs' && e.position.distanceTo(bot.entity.position) < config.hostileRadius
      )
      if (hasHostile) {
        clearTimeout(this._hostileDebounceTimer)
        this._hostileDebounceTimer = setTimeout(() => this.emit('hostileNearby'), config.hostileDebounceMs)
      }
    }

    bot.on('health', onHealth)
    bot.on('time', onTime)
    bot.on('entitySpawn', onEntitySpawnOrMoved)
    bot.on('entityMoved', onEntitySpawnOrMoved)

    return () => {
      bot.off('health', onHealth)
      bot.off('time', onTime)
      bot.off('entitySpawn', onEntitySpawnOrMoved)
      bot.off('entityMoved', onEntitySpawnOrMoved)
      clearTimeout(this._hostileDebounceTimer)
    }
  }
}
```

### Padrão 4: nextMechanicalStep — arbiter-as-classifier com caminho null

```typescript
// [ASSUMED] — baseado na leitura de arbiter.ts existente
export interface MechanicalStep {
  skill: string
  target: string
}

export function nextMechanicalStep(
  snapshot: WorldSnapshot,
  holder: CognitiveStateHolder,
  safety: SafetyState,
  outcome: SkillOutcome | null, // outcome do último SkillResult
): MechanicalStep | null {
  // Outcome-driven: no_effect/partial repetido → escala ao LLM (D-07)
  if (outcome === 'no_effect' || outcome === 'partial') {
    if (shouldAbandon(safety)) return null // força LLM
  }

  // Caminho mecânico: arbiter determina o estado
  const mode = holder.control.getMode()
  if (mode === 'paused') return null // controle externo → LLM decide
  const excluded = cooledDownTargets(safety, Date.now())
  const gatherTarget = highestPriorityGatherTarget(snapshot, excluded)
  if (gatherTarget) {
    return { skill: 'dig', target: gatherTarget }
  }

  // null = "não há passo mecânico óbvio → escala ao LLM"
  return null
}
```

### Padrão 5: emit('actionFinished') após cada skill no execute

O nó `execute` já tem o resultado antes de retornar. A emissão do `actionFinished` deve acontecer DENTRO do nó execute (ao final do try/catch/finally), com o `SkillResult` como payload, antes do `return { memory: holder.memory }`. O driver recebe o evento via race e dispara imediatamente a próxima iteração do grafo.

**Ponto crítico:** o `actionFinished` deve ser emitido DEPOIS do grounding (captura do `after` e avaliação do delta) mas ANTES de retornar do nó. O driver não tem acesso ao `SkillResult` diretamente — apenas o evento acorda o race.

### Anti-Patterns a Evitar

- **Assinar `bot.on('time', ...)` cru:** dispara 20×/s; usar edge-detection com `_wasDay` (D-13).
- **Assinar `bot.on('entityMoved', ...)` cru para hostileNearby:** idem; usar debounce de 500–1000ms.
- **Manter `await sleep(minTickMs)` como fallback:** qualquer sleep fixo reintroduz o tick disfarçado (D-13). O timeout-piso do race é rede de segurança, não motor.
- **Abortar via `throw` no AbortSignal sem `finally`:** o grounding pós-skill deve sempre rodar (Fase 7 existente). Usar `try/catch/finally` como hoje.
- **Não limpar listeners do TriggerBus no `bot.end`:** vazamento de EventEmitter; o `cleanup()` retornado por `setupMineflayerListeners` deve ser chamado no `bot.once('end')`.
- **Chamar `emit('actionFinished')` antes do grounding:** o roteador precisa do `holder.lastObservedDelta` preenchido para decidir mecânico vs LLM (D-07).

## Don't Hand-Roll

| Problema | Não construir | Usar | Por que |
|----------|---------------|------|---------|
| Classificação de mob hostil | Lógica de nomes custom ("zombie", "creeper", ...) | `entity.kind === 'Hostile mobs'` (já no Mineflayer via minecraft-data) | Cobre todos os 1.21.4 hostis automaticamente; lista nomes é frágil com updates |
| Cancelamento de pathfinder | Promise que rejeita via flag custom | `bot.pathfinder.stop()` (API existente) | Já integrado ao `goto`; emite `path_stop` que rejeita com `PathStopped` |
| Timeout/watchdog de skill | Re-implementar no nó execute | `executeWithSafety` existente (executor.ts) | Já tem `Promise.race` + watchdog; AbortSignal é o 4° racer |
| Debounce de eventos | `setTimeout` inline em cada handler | Closure no TriggerBus com `clearTimeout` antes de novo `setTimeout` | Padrão canônico; evita acúmulo de timers |
| Park/wake com EventEmitter | Polling com `setTimeout` fixo | `Promise.race` com `emitter.once` | Elimina CPU ocioso; o evento resolve imediatamente |

## Common Pitfalls

### Pitfall 1: emit('actionFinished') antes do grounding completo

**O que dá errado:** Se o TriggerBus emitir `actionFinished` antes de `holder.lastObservedDelta` ser preenchido, o driver dispara a próxima iteração do grafo com o delta da ação ANTERIOR — o roteador D-07 lê outcome errado.

**Por que acontece:** O nó `execute` faz captura de `after` e avaliação do delta após o `try/catch`; é tentador emitir o evento no `try`, antes do `finally`.

**Como evitar:** Emitir `actionFinished` como a ÚLTIMA coisa antes do `return { memory: holder.memory }` no nó `execute`, após `holder.lastObservedDelta` já estar preenchido.

### Pitfall 2: Race condition entre o abort e o grounding

**O que dá errado:** O AbortSignal dispara, a skill rejeita com `AbortError`, mas o `finally` do nó `execute` não captura o after-state corretamente porque o bot está no meio de uma animação.

**Por que acontece:** `bot.pathfinder.stop()` para o pathfinder imediatamente mas o bot físico pode estar em movimento por alguns ticks.

**Como evitar:** O grounding (`captureGroundState`) é capturado no `catch` do nó execute (pattern já existente em dig.ts/navigate.ts — `caught + evaluateDig/Navigate` roda sempre). O AbortError deve ser tratado como qualquer outro erro de skill pelo grounding.

### Pitfall 3: Vazamento de listeners do TriggerBus

**O que dá errado:** A cada reconexão de bot, novos listeners são adicionados ao TriggerBus sem remover os antigos. Após 3 reconexões, cada evento de fome dispara 3× os handlers.

**Por que acontece:** O `startCognitiveLoop` é chamado a cada reconexão com o mesmo `holder`. Se o TriggerBus for instanciado fora do holder (correto per D-12) mas os handlers do `bot` não forem limpos, acumulam.

**Como evitar:** `setupMineflayerListeners` retorna uma função `cleanup()`. Chamar `cleanup()` no `bot.once('end', ...)` handler (D-11). Criar novo TriggerBus a cada sessão (instanciado no driver ao lado dos handlers de `bot.on('chat')`).

### Pitfall 4: Timeout-piso muito baixo vira tick disfarçado

**O que dá errado:** Se `nextWakeMs` padrão (o valor emitido pelo grafo quando `!enteredIdle`) for muito baixo (ex: 500ms = `minTickMs` atual), o agente que não recebe `actionFinished` (bug, skill travada) desperta a cada 500ms — o tick disfarçado retorna.

**Por que acontece:** O timeout-piso é rede de segurança mas se tornar a rota principal, nega o objetivo da fase.

**Como evitar:** O `nextWakeMs` para o estado de "aguardando skill" deve ser alto (ex: `navigateTimeoutMs` = 30s) — tempo máximo de uma skill. Para o estado idle genuíno, o agente sugere via D-10 um `nextWakeMs` confortável (ex: 60s). Documentar isso na config.

### Pitfall 5: `bot.time` dispara `'time'` event a 20Hz

**O que dá errado:** O handler `bot.on('time', ...)` no TriggerBus é chamado 20 vezes por segundo (tick do servidor) mesmo quando só queremos detectar a transição dia→noite (~1×/dia de jogo = ~10min reais).

**Por que acontece:** `bot.time.timeOfDay` atualiza a cada tick do servidor; o Mineflayer emite `'time'` a cada atualização.

**Como evitar:** Usar `_wasDay` flag + `if (isDay !== this._wasDay)` antes de emitir. Custo: uma comparação booleana por tick. Nenhum timer necessário.

### Pitfall 6: AbortSignal orphan no nó execute

**O que dá errado:** O `skillAbort.abort()` no `finally` é chamado mesmo quando a skill terminou normalmente — isso dispara o listener `onHostileNearby` que foi registrado para abortar, mas o listener já foi removido. Sem listener, `abort()` é no-op. OK. MAS se o listener não for removido no `finally`, ele permanece no TriggerBus para a próxima skill.

**Como evitar:** Sempre remover o listener no `finally` via `triggerBus.off(eventName, handler)` ANTES de chamar `skillAbort.abort()`.

## Open Questions (RESOLVED)

1. **Onde emitir `actionFinished` quando não há skill (estado idle/sem ação)?**
   - O que sabemos: o nó execute retorna sem chamar nenhuma skill em `idle`/`building(stub)`/`fighting(stub)`.
   - O que é incerto: nesse caso, o driver deve disparar a próxima iteração imediatamente ou esperar o timeout-piso?
   - Recomendação: emitir `actionFinished` mesmo quando `skill === null` — a ação "decidir não fazer nada" também termina, e o grafo deve rodar imediatamente para reavaliar. O timeout-piso só importa quando o park realmente estacionar (enteredIdle=true).

2. **Como tratar `stuck` como evento do TriggerBus (D-14)?**
   - O que sabemos: `SkillStuckError` é lançado pelo watchdog no `executeWithSafety`. Hoje é capturado no `catch` do execute e gravado como `outcome:'error'`.
   - O que é incerto: o `stuck` deve ser emitido ANTES ou DEPOIS do grounding?
   - Recomendação: emitir `stuck` no `catch` do nó execute ANTES do emit(`actionFinished`), mas APÓS `holder.lastObservedDelta` ser preenchido. O driver recebe ambos em sequência; o `actionFinished` é o gatilho de loop; o `stuck` é informação para o roteador e futura Phase 8.

3. **O `bot.collectBlock.collect` cancela ao `pathfinder.stop()`?**
   - O que sabemos: `collectBlock.collect` usa pathfinder internamente (via `movements`). `pathfinder.stop()` seta `stopPathing = true`.
   - O que é incerto: se `collectBlock` tem seu próprio loop que ignora `stopPathing`.
   - Recomendação: verificar em runtime. Se não cancelar, o AbortSignal no `executeWithSafety` já encerra a Promise (o `collectBlock` fica como "ghost action" por um curto intervalo mas o executor já parou de esperar). Aceitar como limitação do v1; a Phase 8 pode endereçar.
