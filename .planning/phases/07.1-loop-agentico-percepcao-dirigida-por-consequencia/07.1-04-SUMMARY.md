# 07.1-04 — AbortController/AbortSignal: preempção de skills por evento

## O que foi feito

Adicionado suporte completo a `AbortSignal` como mecanismo de preempção entre o nó `execute` e as skills. Quando um evento `hostileNearby` chega no TriggerBus enquanto uma skill está em execução, o bot para o pathfinder imediatamente em vez de aguardar até a skill terminar.

## Mudanças por arquivo

### `src/skills/executor.ts`
- Adicionado campo `signal?: AbortSignal` à interface `ExecuteOptions` (D-16)
- Adicionado `abortPromise` como 4° racer no `Promise.race` dentro de `executeWithSafety`
- A promise rejeita com `Error('AbortError')` quando o sinal é disparado (ou já estava abortado)

```typescript
export interface ExecuteOptions {
  timeoutMs?: number
  progressChecker?: ProgressChecker
  progressIntervalMs?: number
  noProgressToleranceMs?: number
  /** D-16: AbortSignal para preempção externa da skill (4° racer no race). */
  signal?: AbortSignal
}
```

### `src/skills/navigate.ts`
- Extração de `signal` de `rawParams` ANTES do `NavigateSchema.parse` (signal não é serializável/Zod)
- Handler `signal.addEventListener('abort', () => bot.pathfinder.stop(), { once: true })` registrado antes do `executeWithSafety`
- `signal` passado para `executeWithSafety` como 4° racer (D-17)

```typescript
// Extração antes do Zod:
const signal = (rawParams as Record<string, unknown>)?.signal as AbortSignal | undefined
const { target, range } = NavigateSchema.parse(rawParams)

// Handler de abort:
if (signal) {
  signal.addEventListener('abort', () => {
    try { bot.pathfinder.stop() } catch { /* ignora se pathfinder já parou */ }
  }, { once: true })
}
```

### `src/skills/dig.ts`
- Mesmo padrão de navigate: extração de `signal` antes do Zod, handler `pathfinder.stop()` no abort
- `signal` passado para os dois `executeWithSafety` (ramo `collectBlock.collect` e ramo `bot.dig`)
- Limitação aceita v1: `collectBlock` pode continuar por alguns ticks após `pathfinder.stop()`, mas o executor para de aguardar a Promise imediatamente (Open Question 3 da research)

### `src/cognition/nodes.ts` — nó `execute`
- `AbortController` criado por skill-run (D-18)
- Listener `triggerBus.once('hostileNearby', onHostileNearby)` registrado ANTES de chamar a skill
- `skillAbort.signal` injetado nos params da skill (`{ target, signal: skillAbort.signal }`)
- Bloco `finally` garante ordem crítica (Pitfall 6): `triggerBus.off` → `skillAbort.abort()` (idempotente)
- `triggerBus.emit('actionFinished', ...)` movido para APÓS o `finally` (fora do catch — Pitfall 1)

```typescript
const skillAbort = new AbortController()
const onHostileNearby = () => {
  log(`preemptando ${skill} — hostileNearby recebido`)
  skillAbort.abort('hostileNearby')
}
triggerBus.once('hostileNearby', onHostileNearby)
try {
  // ... skill com signal: skillAbort.signal ...
} catch (err) {
  // grounding no catch — continua mesmo após AbortError
} finally {
  triggerBus.off('hostileNearby', onHostileNearby)  // remove listener ANTES do abort
  skillAbort.abort()  // cleanup idempotente
}
triggerBus.emit('actionFinished', { skill, outcome: skillOutcome })
```
