# Phase 08.1: Refatorar memória — Research

**Researched:** 2026-06-21
**Domain:** Memória de agente persistente — vector store HTTP (ChromaDB v3) + SQLite relacional (bun:sqlite) + fiação do loop cognitivo
**Confidence:** HIGH (API do Chroma e código existente verificados; constantes/granularidade são discricionárias por design)

## Summary

A memória de longo prazo do MineMind está "praticamente morta" porque o caminho de escrita nunca é chamado em produção: o código emite `push(holder.memory, …)` nos pontos de origem reais do loop (`loop.ts:125`, `nodes.ts:238/302/322`), mas `persistEvent` (longTerm.ts) só aparece em testes. Confirmado por grep: **zero** chamadas de `persistEvent` fora de `*.test.ts`. Esse é o bug de fiação central (D-06/D-08). A correção é um helper `recordEvent(holder, db, e, now)` = `push` + `persistEvent(embedding=null)`, plugado onde hoje só há `push`.

A migração vetorial sai de `sqlite-vec` (virtual table `vec0` no mesmo arquivo) para **ChromaDB v3 via HTTP**. O cliente `chromadb@3.4.3` (npm, abril/2026) é **JS puro** — sua única dependência runtime é `semver` (verificado via `npm view`), então roda no Bun sem node-gyp/NAPI (espelha a folga "fetch-puro" que a STACK já reconhece para o Bun). A topologia é **híbrida** (D-01): SQLite continua source of truth (`events`/`players`/`places`/`kv`/`lessons`); o Chroma é só índice derivado e descartável, reconstruível por re-embed dos `events`. Toda escrita/leitura no Chroma passa por **circuit breaker + timeout de fetch** (D-02/D-22) que degrada para o fallback de recência que `retrieve` já implementa — o loop nunca aborta.

O terceiro pilar é **uso real pelo LLM**: hoje `serializeContext` no caminho de AÇÃO (deliberation.ts) **não injeta nenhuma memória recuperada**. A correção é passar `recalled` (top-k=3) com query = embedding do `currentGoal` (cacheado por hash) e renderizar uma seção `Memórias relevantes:` no HumanMessage, antes do `FATO VERIFICADO`. Soma-se a isso memória espacial (POIs em `places` com coluna `type` + busca por bounding-box), registro de morte (evento `type:'death'` importância 10 + danger POI) e a semente de `lessons` (só storage + reforço/decay + consulta).

**Primary recommendation:** Comece pelo bug de fiação (`recordEvent`) — é a maior alavanca e independe do Chroma. Em paralelo, encapsule o Chroma atrás de `src/memory/chromaClient.ts` com breaker/health-check, mantendo `persistEvent`/`retrieve` como a única superfície que o resto do código vê. Use `chromadb@3.4.3`, `getOrCreateCollection({ embeddingFunction: null, configuration: { hnsw: { space: 'cosine' } } })`, e bring-your-own-embeddings via `provider.embed` (que já é sempre local).

### Pattern 1: ChromaClient com bring-your-own-embeddings + get-or-create (D-03/D-05/D-23)
**What:** Conectar a `localhost:8000`, garantir a collection `events` (sem embedding function, cosine, dim implícita 768 na 1ª inserção).
**When to use:** No boot (`chromaClient.ts`), antes de qualquer `add`/`query`.
```typescript
// Source: https://docs.trychroma.com/docs/collections/configure + manage-collections (chromadb 3.4.3)
import { ChromaClient } from 'chromadb'

// Default já é http://localhost:8000, default_tenant/default_database.
const client = new ChromaClient({ host: 'localhost', port: 8000, ssl: false })

// get-or-create (D-23): cria se não existir, ignora args se já existe.
// embeddingFunction: null = bring-your-own (D-03). configuration.hnsw.space cosine (D-05; default é l2!).
const collection = await client.getOrCreateCollection({
  name: 'events',
  embeddingFunction: null,
  configuration: { hnsw: { space: 'cosine' } },
})
```

### Pattern 2: add / query bring-your-own-embeddings (D-04)
**What:** Inserir vetores já computados; consultar por `queryEmbeddings` com filtro `where` por metadata.
```typescript
// Source: https://docs.trychroma.com/docs/querying-collections/query-and-get
// ADD — id deve ser STRING (use String(eventId) do SQLite). metadatas plano (string|number|bool).
await collection.add({
  ids: [String(eventId)],
  embeddings: [embedding],            // number[][] — uma linha por id
  metadatas: [{ type: e.type, player: player ?? '', ts: e.timestamp, importance }],
  documents: [summary],               // o summary determinístico (summarizeEvent)
})

// QUERY — where cobre o filtro por player que hoje é JOIN (D-04).
const res = await collection.query({
  queryEmbeddings: [queryEmbedding],  // number[][]
  nResults: config.retrievalK,        // 12 candidatos; o top-k=3 é cortado depois no scoring
  where: player ? { player: { $eq: player } } : undefined,
})
// RETORNO é DOUBLE-NESTED por query: res.ids[0], res.distances[0], res.documents[0], res.metadatas[0].
// v3: embeddings NÃO vêm por padrão em query/get (mudança de default). distances vêm.
const ids = res.ids[0]            // string[]
const distances = res.distances[0] // number[]  (cosine distance ∈ [0,2]; relevance = clamp(1 - dist))
```

### Pattern 3: Circuit breaker leve com AbortController (D-02/D-22)
**What:** closed → open (após N falhas) → half-open (após cooldown) → closed (sucesso). Timeout no fetch via `AbortSignal.timeout`. Log debounced. Religa sozinho.
**When to use:** Envolve TODA chamada ao Chroma (`add`/`query`/`heartbeat`).
```typescript
// Espelha o padrão openDb graceful (persistence.ts): degrada, nunca lança.
type BreakerState = 'closed' | 'open' | 'half-open'

interface Breaker {
  state: BreakerState
  failures: number
  openedAt: number
  lastWarnAt: number
}

const COOLDOWN_MS = 30_000      // discricionário (D-02) — quando religar (half-open)
const FAIL_THRESHOLD = 3        // discricionário
const FETCH_TIMEOUT_MS = 2_000  // discricionário — timeout curto p/ não pendurar o tick
const WARN_DEBOUNCE_MS = 60_000 // D-22 — não inunda o log

async function withBreaker<T>(b: Breaker, fn: () => Promise<T>, fallback: T): Promise<T> {
  const now = Date.now()
  if (b.state === 'open' && now - b.openedAt < COOLDOWN_MS) {
    warnOfflineDebounced(b, now) // D-22
    return fallback              // caminho vetorial desligado, loop segue (D-02)
  }
  if (b.state === 'open') b.state = 'half-open' // tenta religar
  try {
    // AbortSignal.timeout (Bun/Node 18+) — corta a chamada pendurada.
    const out = await fn()       // fn usa { signal: AbortSignal.timeout(FETCH_TIMEOUT_MS) } no fetch interno
    b.state = 'closed'; b.failures = 0
    return out
  } catch (err) {
    b.failures++
    if (b.failures >= FAIL_THRESHOLD) { b.state = 'open'; b.openedAt = now }
    warnOfflineDebounced(b, now)
    return fallback
  }
}
```
> **Nota:** o cliente `chromadb` aceita `fetchOptions` no construtor; passe um `signal` lá OU envolva cada chamada num `Promise.race` com timeout. `AbortSignal.timeout(ms)` é suportado no Bun. Confirmar na implementação que o cliente repassa o `signal` (caveat MEDIUM — se não repassar, use `Promise.race`).

### Pattern 7: Busca por proximidade (bounding-box + euclidiana) (D-16)
```typescript
const R = 64 // raio de busca em blocos (discricionário)
const rows = db.prepare(`
  SELECT key, label, type, x, y, z, visits FROM places
  WHERE x BETWEEN ? AND ? AND z BETWEEN ? AND ?
`).all(px - R, px + R, pz - R, pz + R) as PlaceRow[]
// ordena euclidiano no JS (top-N) — barato p/ o nº de POIs esperado.
const nearby = rows
  .map(r => ({ ...r, dist: Math.hypot(r.x - px, r.y - py, r.z - pz) }))
  .sort((a, b) => a.dist - b.dist)
  .slice(0, N)
// render no prompt: "POIs próximos: base (12m), veia de ferro (40m)"
```

### Anti-Patterns to Avoid
- **Embed no `push`/tick:** viola "tick nunca espera o LLM" (D-07). O único embed é o batch da reflexão.
- **`await` no Chroma dentro do tick rápido:** mesmo com breaker, mantenha as escritas de vetor no ritmo lento (reflexão); o `add` de evento individual grava embedding **null** (D-06) — não chama o Chroma no caminho quente.
- **Tratar o Chroma como source of truth:** se o `add` falhar, o evento JÁ está no SQLite; o índice fica stale, não corrompido (D-01). Nunca bloquear a escrita relacional por falha de vetor.
- **Default de espaço errado:** o default do Chroma é **`l2`**, não cosine — esquecer `configuration.hnsw.space='cosine'` produz scoring silenciosamente errado (D-05).
- **Ids numéricos no Chroma:** `ids` são **string**; converta `String(eventId)`.
- **Confiar em `embeddings` no retorno de `query`:** v3 mudou o default — `query`/`get` **não** incluem embeddings. Use `distances` (que vêm) para a relevância.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Protocolo HTTP do Chroma (rotas v2, serialização) | `fetch` cru aos endpoints | Cliente `chromadb@3.4.3` | Já é fetch-puro, tipado e mantido; reimplementar é só superfície de bug. |
| Scoring de relevância/recência/importância | Novo ranqueador | `retrieve` (longTerm.ts) já faz Generative Agents min-max | Reusar — só trocar a fonte do KNN (vec0 → Chroma) e passar `queryEmbedding`. |
| Heurística de importância e summary NL | Novo classificador | `importanceOf`/`summarizeEvent` (já existem) | D-08/D-18 só estendem (adicionar `death`=10). |
| Embedding HTTP a LM Studio | Novo cliente | `provider.embed` / `createLocalEmbedder` (já existem) | Sempre local (PROV-03), já tratado. |
| Degradação graciosa de DB | Novo wrapper | Espelhar `openDb` (persistence.ts) | O padrão de recovery já está provado; o breaker do Chroma é a versão "de rede" dele. |

**Key insight:** O circuit breaker em si **deve** ser hand-rolled (lib `opossum` é Node-cêntrica e excessiva para D-02) — mas é a ÚNICA peça nova; todo o resto (scoring, importância, embed, recovery) já existe e só precisa de re-fiação. O risco da fase é de **integração**, não de algoritmo.

## Common Pitfalls

### Pitfall 1: API da série 3.x mudou entre minors
**What goes wrong:** Exemplos de v1/v2 (e até 3.0.x) usam `metadata: {"hnsw:space": "cosine"}` ou `new ChromaClient({ path })`; a v3.4.x usa `configuration: { hnsw: { space } }` e construtor por `{ host, port, ssl }`.
**Why it happens:** Treino e blogs majoritariamente pré-v3; default mudou de `l2`.
**How to avoid:** Pinar `3.4.3`; usar `configuration.hnsw.space` (não metadata); validar o objeto retornado por `getOrCreateCollection` no boot. Ler o CHANGELOG antes de qualquer upgrade.
**Warning signs:** `getCollection` exigindo `embeddingFunction` (bug #3730 do JS client — em versões antigas era required em `getCollection`; em `getOrCreateCollection` é opcional). Scoring "estranho" = espaço l2 esquecido.

### Pitfall 2: Dimensão travada na 1ª inserção
**What goes wrong:** A collection fixa a dimensão no 1º `add`. Se o 1º vetor tiver dim ≠ 768 (modelo de embedding trocado), todo `add`/`query` futuro com 768 falha.
**Why it happens:** Modelo de embedding do LM Studio mudou sem recriar a collection.
**How to avoid:** Validar `embedding.length === EMBEDDING_DIM` antes do `add` (já é feito em `persistEvent`/`consolidate`); D-05 valida no boot. Se a dimensão divergir, dropar/recriar a collection (é descartável — D-01).
**Warning signs:** Erro de dimension mismatch do Chroma; índice "stale" persistente.

### Pitfall 3: O bug de fiação reaparece se `recordEvent` não cobrir TODOS os pontos
**What goes wrong:** Plugar `recordEvent` só em um ou dois `push` deixa eventos sem persistir — `events` continua subpovoado.
**Why it happens:** Há **4** pontos de origem: `loop.ts:125` (reflexo idle), `nodes.ts:238/302/322` (execute/observe). O grep desta pesquisa confirma exatamente esses.
**How to avoid:** Substituir cada `push(holder.memory, …)` por `recordEvent(holder, db, …)` (que faz o push internamente). Verificar ao vivo que `SELECT count(*) FROM events > 0` após uma sessão.
**Warning signs:** `events=0` após rodar; `[recall]` nunca acha nada.

### Pitfall 4: Reflexão nunca dispara (Known Gap da Fase 4) → `vec_events`/vetores vazios
**What goes wrong:** Como o único embed é a reflexão (D-07), se `shouldReflect`/`consolidate` não roda ao vivo, o Chroma fica vazio mesmo com `events` povoado.
**Why it happens:** `inFlight` da deliberação ocupado; `enteredIdle` mal cabeado; System 1 (Fase 8) mudou quando o lock fica livre (regressão B1 sinalizada no STATE).
**How to avoid:** D-09 — confirmar o disparo ao vivo (log `[reflect] reflexão executada` já existe em loop.ts:365). Re-testar pós-System 1 (blocker registrado no STATE).
**Warning signs:** `events` cresce mas nenhum `type:'reflection'` aparece; Chroma collection com count 0.

### Pitfall 5: Chroma pendurado (não offline) trava o tick
**What goes wrong:** Servidor responde lento (não recusa conexão) → `fetch` sem timeout pendura.
**Why it happens:** Breaker só conta falhas, não trata "lento".
**How to avoid:** `AbortSignal.timeout(FETCH_TIMEOUT_MS)` em toda chamada; timeout conta como falha → abre o breaker (Pattern 3). Manter o embed/vetor no ritmo lento, nunca no tick.
**Warning signs:** Ticks longos esporádicos correlacionados com I/O do Chroma.

### Pitfall 6: `where` com tipos inválidos / id numérico
**What goes wrong:** metadata com `null`/`undefined` ou `ids` numéricos → erro de validação (v3 tem ValidationException tipada).
**Why it happens:** `player` pode ser null no SQLite; ids são INTEGER no SQLite.
**How to avoid:** Coagir metadata para `string|number|boolean` (use `player ?? ''`), `ids: [String(eventId)]`. Filtrar `where` só quando `player` definido.
**Warning signs:** `ValidationException` no `add`/`query`.

## Open Questions

1. **O cliente `chromadb@3.4.3` repassa o `signal` (AbortController) do `fetchOptions` para todas as chamadas?**
   - What we know: o construtor aceita `fetchOptions`; o cliente é fetch-puro.
   - What's unclear: se um `signal` em `fetchOptions` cobre cada `add`/`query` ou só o handshake. (MEDIUM)
   - Recommendation: na implementação, verificar empiricamente; se não cobrir, envolver cada chamada num `Promise.race([fn(), timeout()])`. O breaker funciona em qualquer caso.

2. **Reconciliação de ids entre SQLite (INTEGER autoincrement) e Chroma (string id).**
   - What we know: `events.id` é a chave; o Chroma exige id string.
   - What's unclear: nada de bloqueante — `String(id)` resolve. Mas o `add` do vetor acontece na reflexão (D-07), depois do INSERT relacional do consolidate; garantir que o id usado no Chroma é o `lastInsertRowid` do evento consolidado.
   - Recommendation: gravar o vetor no Chroma usando `String(idConsolidado)` dentro de `consolidate` (ou logo após), não no `persistEvent` do evento individual (que grava embedding null).

3. **Quando reconstruir o índice (re-embed) — boot, comando, ou nunca automático?**
   - What we know: D-01 torna o índice reconstruível; events=0 hoje.
   - What's unclear: se a reconstrução roda no boot quando detecta collection vazia mas events>0 (índice perdido).
   - Recommendation: helper `migrate-vectors.ts` one-shot manual nesta fase (não automático no caminho quente); automação fica para evolução.
