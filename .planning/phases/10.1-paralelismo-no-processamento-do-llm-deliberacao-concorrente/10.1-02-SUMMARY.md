# Phase 10.1 Plan 02: Fiação da Concorrência no Loop Cognitivo Summary

**Substitui o lock single-flight `inFlight` por gate-por-tipo + semáforo global (`= provider.maxConcurrency`) no loop, roteia `handleConversation` pelo mesmo gate (prioridade player=0) com preempção da AÇÃO via AbortController, e blinda `holder.goals` do clobber concorrente via commit síncrono merge-by-id na reflexão — a entrega comportamental da fase.**

## Accomplishments

### Task 1 — `deliberation.ts`: gate-por-tipo + semáforo, commit síncrono merge-by-id (D-01/D-04/D-05/D-13)

- **`DeliberationState`** perde o `inFlight: boolean` único — resta só `lastRunAt` (orçamento de replan de AÇÃO). O controle de sobreposição passa a ser o `gate` por tipo + o `semaphore`, injetados em `maybeDeliberate`.
- **`maybeDeliberate`** ganha 3 params (`gate`, `semaphore`, `signal?`, com defaults para retrocompat dos testes). Mapeia `trigger → TaskType` (`reflect→reflection`, demais→`action`) e `priority` (reflection=2, action=1). Fluxo: `gate.tryEnter(taskType)` (no-op se o MESMO tipo já roda) → orçamento de replan SÓ para ação → `shouldTrigger` → `semaphore.acquire(priority)` → `try { ... } finally { semaphore.release(); gate.leave(taskType) }`. **release/leave SEMPRE no finally** (Pitfall 3).
- **`runReflection`** documenta e garante o **commit 100% síncrono** (D-04/D-05): do `holder.goals = applyGoalUpdates(holder.goals, updates, now)` até o `persistHolder` NÃO há `await`. O re-read de `holder.goals` no instante do write + o merge-by-id de `applyGoalUpdates` fazem um goal de `player_request` empurrado DURANTE o await da reflexão **sobreviver** ao commit (Pitfall 2 fechado). A reflexão **nunca recebe signal** (D-13).
- **`decideAction`** (structured.ts) ganha `opts?: { signal? }` e repassa a `provider.decide` nas duas chamadas (1ª tentativa + repair) — habilita a preempção da AÇÃO.
- **`createDeliberator`** remove `inFlight: false` do state inicial; a assinatura retornada inclui os novos params opcionais.

### Task 2 — `loop.ts`: fiação por sessão, roteamento do chat, preempção (D-08/D-11/D-12/D-13)

- **Instanciação por sessão:** `new Semaphore(provider.maxConcurrency)` + `createTaskGate()` logo após `createDeliberator()`; `let actionAbort: AbortController | null = null`.
- **`routePlayerTurn`** (helper puro exportado): `if (!gate.tryEnter('player')) return` → `preemptAction()` → `await semaphore.acquire(0)` → `try { run() } finally { release(); leave('player') }`. O handler de chat troca `void handleConversation(...)` por `void routePlayerTurn(llmSemaphore, taskGate, () => actionAbort?.abort(), () => handleConversation(...))` — fecha a brecha do chat-sem-coordenação (D-08).
- **Preempção da AÇÃO (D-12):** o dispatch de ação cria `const ctrl = new AbortController(); actionAbort = ctrl` e passa `ctrl.signal` a `maybeDeliberate`; `.finally(() => { if (actionAbort === ctrl) actionAbort = null })` limpa só o controller deste dispatch (evita corrida). O ramo de REFLEXÃO **não** cria nem passa AbortController (D-13).
- **`pickDispatch` não-XOR:** vira `{ reflect, action }` booleanos independentes (`reflect = reflectDue && !reflectionBusy`; `action = !actionBusy`). Reflect e action podem ambos ser despachados no mesmo tick — o gate por tipo + o semáforo coordenam (Pitfall 6).
- **`shouldPreemptAction(hasPlayerTurn, actionInFlight)`** — predicado puro auxiliar.

## Caveats de Validação AO VIVO (HUMAN-UAT, não-bloqueante)

A entrega é comportamental e coberta por testes determinísticos, mas três pontos só se confirmam com LM Studio + servidor MC ao vivo:

1. **LM Studio `Max Concurrent Predictions` (D-09):** com `maxConcurrency` local = 4 (config do Plan 01), a coexistência real ação×reflexão depende de o LM Studio ter `Max Concurrent Predictions ≥ 2` habilitado (default 4). Com 1, degrada ao single-flight priorizado + preempção (D-03) — ainda correto, sem ganho de sobreposição.
2. **`RunnableConfig.signal` aborta o fetch local (D-14/D-15):** o Plan 01 confirmou a propagação por testes; ao vivo, validar que abortar a AÇÃO via `actionAbort.abort()` de fato cancela o fetch do LM Studio e libera o slot (no cloud o servidor segue gerando — D-15, aceito). Não há uso de `AbortSignal.any` manual (o LangChain compõe com o timeout interno), então o Pitfall 1 (Bun sem `AbortSignal.any`) não se aplica a este plano.
3. **`[reflect]` ao vivo + preempção jogador→ação:** re-teste limpo confirmando que (a) a reflexão dispara e não starva sob carga de ação, e (b) um pedido de jogador preempta uma AÇÃO em voo e a ação re-despacha no próximo tick. Liga-se ao Known Gap "`[reflect]` ao vivo não confirmado".
