## User Constraints (from CONTEXT.md)

### Locked Decisions

**Modelo de concorrência (substitui o `inFlight` único):**
- **D-01:** Substituir `inFlight: boolean` por **gate por tipo de tarefa** (AÇÃO / REFLEXÃO / RESPOSTA-A-JOGADOR — "não sobrepor o mesmo tipo") **combinado com um semáforo global** de tamanho `= provider.maxConcurrency`. O gate por tipo preserva o invariante D-12 (uma inferência por tipo); o semáforo impõe o teto de recurso (GPU/quota).
- **D-02:** Implementação **nativa, zero deps** — primitiva de semáforo hand-rolled (~30 linhas) com `acquire()/release()` e `release()` obrigatório no `finally`. NÃO usar `p-queue`/`p-limit`/`async-mutex`.
- **D-03:** A **"política por capacidade" emerge** do desenho: semáforo tamanho 1 ⇒ single-flight priorizado + preempção; tamanho > 1 ⇒ tarefas distintas sobrepõem. Sem código extra.

**Segurança do estado compartilhado (`holder`):**
- **D-04:** **Commit síncrono** como invariante-base: cada tarefa computa o resultado **fora do `holder`** (durante o `await`) e aplica num **bloco 100% síncrono, sem `await` entre read e write**.
- **D-05:** Conflito real (`holder.goals` escrito pela REFLEXÃO via `applyGoalUpdates` E pela CONVERSA via `goals.push`) resolvido por **merge-reducer**: no commit, a reflexão **re-lê `holder.goals` no instante síncrono e funde por id** em vez de substituir o array. Reaproveita o estilo imutável (`applyGoalUpdates`/`advanceProgress`).
- **D-06:** Precedência de conflito: **goal de jogador sempre vence** drop/reprioritização da reflexão. No pior caso perde-se um delta de reprioritização, **jamais um goal inteiro**.

**Concorrência local vs cloud + teto de custo:**
- **D-07:** Capacidade de concorrência vive na interface `LlmProvider` (`readonly maxConcurrency: number`). O semáforo do driver lê `provider.maxConcurrency`.
- **D-08:** **Todos** os caminhos LLM atravessam o mesmo gate — inclusive `handleConversation` (hoje `void` FORA do `inFlight`).
- **D-09:** LM Studio 0.4.0 aceita requests paralelos via continuous batching (`Max Concurrent Predictions` = 4 default). `maxConcurrency` local deve espelhar ~4 (configurável), NÃO 1.
- **D-10:** **Correção obrigatória — race do `withSpendCap`:** check-then-increment em dois passos abre TOCTOU. Fix: reservar o slot ANTES do disparo (increment-then-check atômico via `INSERT...ON CONFLICT`, com estorno/release no fallback-to-local) OU increment dentro da seção crítica do semáforo cloud.

**Prioridade e preempção:**
- **D-11:** Ordem de prioridade: **jogador > ação > reflexão**.
- **D-12:** Jogador **preempta a deliberação de AÇÃO em voo** via AbortController/AbortSignal (reusa o contrato 07.1 D-16). A ação abortada re-despacha no próximo tick.
- **D-13:** A **REFLEXÃO é protegida nos dois sentidos**: nunca starva (piso `shouldReflect` preservado) e **nunca é abortada** (preserva o flush B2/D-02). Blinda a regressão B1.
- **D-14:** Habilitar preempção exige propagar um `signal` externo por `provider.decide`/`provider.chat` (provavelmente via `RunnableConfig.signal`).
- **D-15:** **Caveat cloud:** abortar o cliente **não interrompe a geração no servidor** OpenAI. A preempção entrega ganho real **só no caminho local**. No cloud, tratar como cancelamento de cliente.

- Valor default do `maxConcurrency` local (≈4) e cloud — via `config`.

## Summary

Esta fase é **concorrência cooperativa pura**, não paralelismo de threads. Node e Bun rodam JS num único thread com event loop; não há data-race de memória — o único hazard é a **janela entre um read e um write quando há um `await` no meio** (D-04). Toda a engenharia da fase gira em torno de manter as seções críticas que tocam o `holder` **100% síncronas** e em mover a propagação de `AbortSignal`/concorrência para a borda do provider.

As três descobertas externas que a CONTEXT.md pediu para verificar estão **confirmadas**: (1) o **LM Studio 0.4.0** processa requests em paralelo via *continuous batching* com `Max Concurrent Predictions = 4` default — logo `maxConcurrency` local deve espelhar ~4, não 1 (D-09 validado); (2) abortar uma chamada OpenAI via `AbortSignal` **não interrompe a geração no servidor** (issue #5001 fechada como "not planned") — a preempção só rende no caminho local (D-15 validado); (3) o caminho de propagação de `signal` no LangChain.js é o campo **`RunnableConfig.signal`** passado como 2º argumento de `invoke` (D-14 validado, com nuance de versão — ver abaixo).

**Nuance crítica de versão (corrige a CONTEXT.md):** o projeto está em **`@langchain/core ^1.2.0` + `@langchain/openai 1.5.1`** (LangChain.js **v1**), não na v0.1 referenciada na base de pesquisa do D-14. A boa notícia: o bug do #5001 (signal não propagado ao OpenAI) foi corrigido nas linhas modernas — `signal` em `RunnableConfig` agora **é** repassado ao fetch da OpenAI. Isso não muda o caveat D-15 (o *servidor* OpenAI continua gerando), mas significa que o `fetch` local do LM Studio é abortado de verdade, liberando o slot do semáforo imediatamente (ganho real de preempção).

**Primary recommendation:** Implementar uma primitiva `Semaphore` hand-rolled (~30 linhas: contador + fila de waiters resolvida por prioridade) e um `Gate` por tipo (3 booleans/locks), instanciados no driver (`loop.ts`) ao lado do `deliberator`. Rotear AÇÃO, REFLEXÃO e `handleConversation` pelo mesmo `acquire(priority)`/`finally release()`. Propagar `AbortSignal` por `provider.decide`/`provider.chat` via `RunnableConfig.signal`, compondo com o `AbortSignal.any([...])` do timeout interno. Tornar `holder.goals` write um **commit síncrono com merge-by-id** (re-lê o array no instante do write). Fechar o TOCTOU do spend-cap com `reserveCall` atômico (`INSERT...ON CONFLICT` com check do `changes`/`RETURNING`) + `releaseCall` no fallback.

## Architecture Patterns

### Estrutura recomendada (onde mora cada peça)
```
src/cognition/
├── concurrency.ts      # NOVO: Semaphore (acquire/release por prioridade) + Gate por tipo
├── loop.ts             # instancia Semaphore(provider.maxConcurrency) + Gate; roteia AÇÃO/REFLEXÃO/CHAT
├── deliberation.ts     # maybeDeliberate: troca inFlight pelo gate+semáforo; commit síncrono
├── reflection.ts       # applyGoalUpdates já merge-by-id; recebe holder.goals ATUAL no commit (D-05)
└── state.ts            # CognitiveStateHolder (sem mudança estrutural; goals continua Goal[])
src/chat/conversation.ts # handleConversation passa pelo gate (D-08); goals.push vira commit síncrono
src/llm/
├── provider.ts         # LlmProvider ganha readonly maxConcurrency; decide/chat ganham { signal? }
├── spendCap.ts         # withSpendCap usa reserveCall/releaseCall (D-10)
└── spendStore.ts       # NOVO: reserveCall (atômico INSERT...ON CONFLICT com check) + releaseCall
```

### Pattern 1: Semáforo assíncrono com prioridade (hand-rolled, D-01/D-02)
**What:** Um contador de permits + uma fila de waiters (Promise resolvers). `acquire()` decrementa se há permit, senão enfileira. `release()` incrementa e acorda o próximo waiter **de maior prioridade**.
**When to use:** Como o teto global de `maxConcurrency` que envolve TODA chamada LLM.
**Padrão idiomático** (resolver-queue, verificado em múltiplas referências — es-toolkit, async-sema, await-semaphore):
```typescript
// src/cognition/concurrency.ts
// Prioridade: menor número = mais urgente (player=0, action=1, reflection=2)
type Waiter = { priority: number; resolve: () => void }

export class Semaphore {
  private permits: number
  private readonly waiters: Waiter[] = []
  constructor(permits: number) { this.permits = Math.max(1, permits) }

  acquire(priority: number): Promise<void> {
    if (this.permits > 0) { this.permits--; return Promise.resolve() }
    return new Promise<void>((resolve) => {
      // inserção ordenada por prioridade (fila estável) — evita starvation por desempate temporal
      const w: Waiter = { priority, resolve }
      const idx = this.waiters.findIndex((x) => x.priority > priority)
      if (idx === -1) this.waiters.push(w); else this.waiters.splice(idx, 0, w)
    })
  }

  release(): void {
    const next = this.waiters.shift()
    if (next) next.resolve()      // permit passa direto ao próximo (não incrementa)
    else this.permits++
  }
}
```
**Uso obrigatório com `finally` (D-02 — evitar permit leak):**
```typescript
await sem.acquire(priority)
try {
  // ... await provider.decide(...)
} finally {
  sem.release()   // SEMPRE, mesmo em throw/abort
}
```
> Nota de design: a "preempção" (D-12) NÃO é o semáforo soltar o slot — é o **AbortSignal** abortar a ação em voo (liberando o slot via `finally`) para que o jogador, já na frente da fila por prioridade, adquira. O semáforo dá a ordem de fila; o abort libera o recurso ocupado.

### Pattern 2: Gate por tipo de tarefa (D-01 — "não sobrepor o mesmo tipo")
**What:** Além do teto global, cada tipo (AÇÃO/REFLEXÃO/CHAT) não pode ter 2 inferências simultâneas **do mesmo tipo** (invariante D-12: modelo local fraco). Três flags substituem o único `inFlight`.
```typescript
type TaskType = 'action' | 'reflection' | 'player'
const gate: Record<TaskType, boolean> = { action: false, reflection: false, player: false }
// tryEnter(type): se gate[type] true → no-op (return false); senão marca e segue.
```
> O `inFlight: boolean` atual (`deliberation.ts` L73, L148, L155, L200) vira `gate.action`/`gate.reflection`. O `pickDispatch` (`loop.ts` L486) deixa de ser "um por tick" e passa a consultar o gate+semáforo.

### Pattern 3: Commit síncrono + merge-by-id (D-04/D-05/D-06) — o coração da segurança
**What:** A tarefa computa o resultado **durante o `await`** (fora do holder) e aplica num bloco síncrono que **re-lê `holder.goals` no instante do write** e funde por id.
**Por que é seguro:** No event loop single-thread, um bloco sem `await` é **atômico por construção** — nenhum outro callback (incluindo `goals.push` da conversa) pode intercalar entre o read e o write. O hazard só existe quando há `await` entre ler e escrever.
**O bug concreto que isto fecha:** hoje `runReflection` faz `holder.goals = applyGoalUpdates(holder.goals, updates, now)` (`deliberation.ts` L278) **após** o `await provider.decide(...)` (L242). Se a CONVERSA fez `holder.goals.push(makePlayerRequestGoal(...))` (`conversation.ts` L160) durante esse `await`, o `applyGoalUpdates` operou sobre uma **cópia stale** de `holder.goals` capturada antes do push → o goal do jogador é **clobberado** ao reatribuir o array.
**Fix:** re-ler `holder.goals` no momento da reatribuição (já síncrono) — `applyGoalUpdates` já é merge-by-id (`reflection.ts` L108 `new Map(updates.map(...))` + L110 loop sobre `goals`), então passar o array ATUAL resolve:
```typescript
// runReflection — em vez de capturar holder.goals antes do await, ler no commit síncrono:
const updates = await computeReflectionUpdates(...)   // await FORA do holder
// ↓ bloco 100% síncrono, sem await daqui até o fim:
holder.goals = applyGoalUpdates(holder.goals, updates, now)  // holder.goals é o ATUAL (pós-push)
if (holder.db) persistHolder(holder.db, holder, now)
```
> `applyGoalUpdates` só faz `drop`/`reprioritize` por id e preserva goals sem update (L112-122). Um goal `player_request:*` empurrado durante o await **não tem id no `updates`** → cai no ramo "inalterado" (L113) e sobrevive. D-06 satisfeito: o goal do jogador nunca some; no pior caso a reflexão perde um delta de reprioritize sobre um goal que ela nem viu.

### Pattern 4: Reserva atômica de slot de custo (D-10 — fecha o TOCTOU)
**What:** `withSpendCap` hoje faz `getCallCount() >= maxCalls` (check) e depois `incrementCall()` (act) em dois passos (`spendCap.ts` L46, L53). Entre os dois há `await provider.decide(...)` — 2 chamadas cloud concorrentes leem `count=N`, ambas passam, ambas incrementam → furam o cap.
**Fix (increment-then-check atômico):** uma operação SQLite única que incrementa e retorna se ainda está sob o teto. Reusa o `INSERT...ON CONFLICT` já presente:
```typescript
// src/llm/spendStore.ts — NOVO
// Reserva 1 slot atomicamente; retorna true se ficou <= maxCalls, false se estourou.
export function reserveCall(db: Database, ts: number, maxCalls: number): boolean {
  ensureSpendTable(db)
  const row = db.prepare(
    `INSERT INTO llm_spend (window_key, calls, tokens) VALUES (?, 1, 0)
     ON CONFLICT(window_key) DO UPDATE SET calls = calls + 1
     RETURNING calls`,
  ).get(windowKey(ts)) as { calls: number }
  return row.calls <= maxCalls   // estourou? caller faz releaseCall + fallback-to-local
}
// Estorna o slot quando caímos no fallback-to-local (não consumiu cloud).
export function releaseCall(db: Database, ts: number): void {
  db.prepare(
    `UPDATE llm_spend SET calls = MAX(0, calls - 1) WHERE window_key = ?`,
  ).run(windowKey(ts))
}
```
```typescript
// withSpendCap.decide — increment-then-check (sem janela TOCTOU):
async decide(schema, messages) {
  const now = Date.now()
  if (store.reserveCall(now, cfg.maxCalls)) {
    try { return await cloud.decide(schema, messages) }
    catch (e) { store.releaseCall(now); throw e }   // erro real: estorna p/ não punir o teto
  }
  store.releaseCall(now)        // estourou: estorna a reserva especulativa
  return local.decide(schema, messages)             // fallback-to-local (custo-zero)
}
```
> ⚠️ Decisão de discrição (CONTEXT.md): "estornar erro real" é opcional — se preferir contar tentativas (não sucessos), não estorne no `catch`. O importante para D-10 é que a **reserva é atômica** e o fallback-to-local **não consome** o slot. `bun:sqlite` é síncrono → o `INSERT...RETURNING` é indivisível no event loop.

### Anti-Patterns a evitar
- **`await` entre read e write do holder:** qualquer `await` entre `const g = holder.goals` e `holder.goals = ...` reabre a janela de clobber. Compute tudo ANTES; só reatribua no bloco síncrono (D-04).
- **`release()` fora do `finally`:** se a tarefa lança/aborta antes do `release()`, o permit vaza e o semáforo seca (deadlock progressivo). Sempre `finally` (D-02).
- **Substituir o array inteiro a partir de cópia stale:** `holder.goals = applyGoalUpdates(staleSnapshot, ...)` clobbera pushes concorrentes. Re-ler no commit (D-05).
- **Confiar que abortar cloud para a cobrança:** falso (D-15). Abort cloud é cancelamento de cliente; o servidor OpenAI segue gerando/cobrando.
- **Preemptar a REFLEXÃO:** proibido (D-13). Só AÇÃO recebe o `signal` de preempção; reflexão roda até o flush B2.

## Don't Hand-Roll

| Problema | Não construa | Use | Por quê |
|----------|--------------|-----|---------|
| Compor signal de preempção + signal de timeout | Encadeamento manual de N `AbortController` com listeners cruzados | `AbortSignal.any([externalSignal, timeoutSignal])` (nativo) | Uma linha; auto-cleanup; aborta se QUALQUER um abortar. (Verificar suporte no Bun alvo — §Pitfalls.) |
| Timeout de chamada LLM | `setTimeout` + `Promise.race` manual em cada call-site | `AbortSignal.timeout(ms)` (nativo) OU o `timeout` já configurado no `ChatOpenAI` (`provider.ts` L107/171) | O `ChatOpenAI` já tem `timeout: 20000`. Compor o signal externo com esse, não reinventar. |
| Contador de custo atômico | Lock manual em JS sobre o contador | `INSERT...ON CONFLICT ... RETURNING` no `bun:sqlite` (síncrono) | SQLite síncrono já é atômico no single-thread; padrão já no projeto (`spendStore.ts` L40). |
| Merge de goals por id | Diff manual de arrays | `applyGoalUpdates` (já existe, já imutável, já merge-by-id) | `reflection.ts` L101-124 — só precisa receber o array ATUAL no commit. |

**Key insight:** A maior parte do "hand-roll" desta fase é **explicitamente travado** (D-02 manda construir o semáforo à mão). Mas a *composição de signals* e a *atomicidade de custo* têm primitivas nativas/já-presentes — não reinvente essas.

## Common Pitfalls

### Pitfall 1: `AbortSignal.any` pode não existir no runtime Bun alvo
**What goes wrong:** `AbortSignal.any([...])` é relativamente recente (Node 20.3+). Se o Bun do projeto for mais antigo, `AbortSignal.any is not a function`.
**Why:** A composição de signals é a forma limpa de unir preempção + timeout, mas é uma API jovem.
**How to avoid:** O Plan deve verificar `typeof AbortSignal.any === 'function'` no ambiente (`bun --version` + teste rápido). Fallback trivial: um `AbortController` que escuta os dois signals (`a.signal.addEventListener('abort', () => c.abort())`), padrão já usado em `navigate.ts` L70.
**Warning signs:** TypeError no boot do primeiro decide com signal.

### Pitfall 2: clobber silencioso de `holder.goals` (a regressão que a fase existe para evitar)
**What goes wrong:** Goal de `player_request` desaparece sem erro — o jogador pede algo, o bot reconhece no chat, mas nunca executa porque a reflexão reatribuiu `holder.goals` por cima.
**Why:** `runReflection` captura `holder.goals` antes do `await` longo do LLM (`deliberation.ts` L242) e reatribui depois (L278); o `goals.push` da conversa (`conversation.ts` L160) cai na janela.
**How to avoid:** Pattern 3 (commit síncrono + re-read). Teste determinístico: simular push concorrente DURANTE o await da reflexão e assertar que o goal sobrevive.
**Warning signs:** "bot reconhece o pedido mas não age" intermitente sob carga de chat.

### Pitfall 3: permit leak → deadlock progressivo do semáforo
**What goes wrong:** Após algumas horas, o bot para de deliberar — todos os permits foram adquiridos e nunca soltos.
**Why:** `release()` não foi chamado num caminho de erro/abort (faltou `finally`).
**How to avoid:** `acquire`/`release` SEMPRE em try/finally (D-02). Teste: forçar throw dentro da seção crítica e assertar que `sem.permits` volta ao valor inicial.
**Warning signs:** deliberação trava sem log de erro; `[deliberate]` para de aparecer mas o loop segue.

### Pitfall 4: throughput local NÃO escala linearmente com `maxConcurrency`
**What goes wrong:** Esperar 4× de throughput ao setar `maxConcurrency=4` no local e frustrar-se.
**Why:** 1 GPU compartilha compute; continuous batching melhora **ocupação**, não multiplica FLOPs (D-09). O ganho local é de **responsividade/concorrência de tarefas** (jogador não espera a ação terminar), não de throughput bruto.
**How to avoid:** Calibrar `maxConcurrency` local ≈ `Max Concurrent Predictions` do LM Studio (4), aceitar ganho sublinear. Throughput real só com cloud.
**Warning signs:** latência por request sobe com concorrência alta no local (batch grande compete por VRAM).

### Pitfall 5: tratar o caveat cloud (D-15) como bug a consertar
**What goes wrong:** Tentar "fazer o abort parar a cobrança da OpenAI" — impossível.
**Why:** O servidor OpenAI não interrompe geração ao receber disconnect (issue #5001, fechada "not planned"). Você cancela o **cliente**, não o **servidor**.
**How to avoid:** No caminho cloud, a preempção é só liberação do slot do cliente (a resposta abortada é descartada). O custo já foi/será cobrado. Aceitar (D-15). Guarda anti-preempção-de-cloud-cobrado é DEFERRED.

### Pitfall 6: o gate-por-tipo virar single-flight de novo
**What goes wrong:** Implementar o gate de forma que AÇÃO e REFLEXÃO ainda se excluam mutuamente (como o `pickDispatch` "um por tick" atual, `loop.ts` L486-489).
**Why:** O `pickDispatch` hoje retorna UM dispatch por tick (`reflect` XOR `action`). Trocar `inFlight` por gate sem soltar essa exclusão preserva o gargalo.
**How to avoid:** Gate é POR TIPO (`gate.action` independente de `gate.reflection`); o semáforo global é quem limita o total. Com `maxConcurrency ≥ 2`, AÇÃO e REFLEXÃO podem coexistir. O `pickDispatch` deixa de ser XOR.

## Code Examples

### Propagar AbortSignal por `decide`/`chat` → `model.invoke` (D-14)
A interface ganha um 3º param opcional; o `ChatOpenAI.invoke` aceita `{ signal }` no 2º arg (`RunnableConfig`), e o `withStructuredOutput(...).invoke` também (é um Runnable):
```typescript
// src/llm/provider.ts — interface
export interface LlmProvider {
  readonly maxConcurrency: number        // D-07
  decide<T>(schema: ZodType<T>, messages: BaseMessage[], opts?: { signal?: AbortSignal }): Promise<T>
  chat(messages: BaseMessage[], opts?: { signal?: AbortSignal }): Promise<string>
  available(): Promise<boolean>
  embed(text: string): Promise<number[]>
}

// decideWithFallback — repassa o signal ao Runnable estruturado:
async function decideWithFallback<T>(model, schema, messages, signal?: AbortSignal): Promise<T> {
  const structured = model.withStructuredOutput(schema, { name: 'decide', method: 'jsonSchema' })
  return (await structured.invoke(messages, { signal })) as T   // RunnableConfig.signal (v1)
}

// chat:
async chat(messages, opts) {
  return String((await model.invoke(messages, { signal: opts?.signal })).content)
}
```
> Fonte: `RunnableConfig` (campo `signal: AbortSignal`) e `BaseLanguageModelCallOptions.signal` — LangChain.js reference. O `ChatOpenAI` já tem `timeout` (`provider.ts` L107/171) que cria seu próprio AbortController interno; o `signal` externo do `RunnableConfig` é **composto** com ele pelo LangChain (ambos abortam a chamada). Não é preciso `AbortSignal.any` manual aqui — mas se quiser compor com signals do projeto, use-o.

### Driver: AÇÃO preemptável + REFLEXÃO protegida (D-12/D-13)
```typescript
// loop.ts — esboço do dispatch concorrente
const actionAbort = new AbortController()   // por dispatch de ação (re-criado a cada ação)

// AÇÃO (preemptável, prioridade 1):
await sem.acquire(1)
try {
  const decision = await deliberateAction({ signal: actionAbort.signal })  // honra abort
  // commit síncrono de holder.llmDecision
} finally { sem.release() }

// Jogador chega (prioridade 0) → preempta a ação em voo:
actionAbort.abort()         // libera o slot da ação via finally; player adquire à frente da fila

// REFLEXÃO (prioridade 2, SEM signal — nunca abortada, D-13):
await sem.acquire(2)
try {
  await runReflection(holder, provider, snapshot, now, chroma)  // sem signal → roda até o flush B2
} finally { sem.release() }
```
> A ação abortada re-despacha no próximo tick (D-12) — o `pickTrigger` (`loop.ts` L491) com `lastOutcome` no_effect/partial já força re-deliberação cedo.

## Open Questions

1. **Versão exata do Bun no ambiente de runtime (suporte a `AbortSignal.any`)**
   - O que sabemos: o `ChatOpenAI` já compõe `timeout` + `RunnableConfig.signal` internamente, então `AbortSignal.any` só é necessário se quisermos unir signals do PRÓPRIO projeto antes de passar.
   - O que falta: confirmar `bun --version` e `typeof AbortSignal.any`.
   - Recomendação: o Plan inclui um check rápido; usa o fallback de `navigate.ts` L70 se ausente.

2. **`maxConcurrency` cloud default**
   - O que sabemos: OpenAI suporta alta concorrência, mas o `withSpendCap` é o limitador econômico real.
   - O que falta: valor default (discrição CONTEXT.md).
   - Recomendação: cloud `maxConcurrency` modesto (ex. 2-3) coordenado com `cloudMaxCallsPerWindow`; o spend-cap é o teto duro, o semáforo é só suavização de burst.

3. **Onde mora o merge-reducer (fatoração — discrição D-05)**
   - O que sabemos: `applyGoalUpdates` (`reflection.ts`) já é merge-by-id imutável.
   - O que falta: passar `holder.goals` ATUAL no commit vs. helper em `state.ts`.
   - Recomendação: manter em `reflection.ts`, apenas garantir que o `runReflection` re-lê `holder.goals` no commit síncrono (mínima mudança; reusa o existente).
