# Phase 11.1: Percepção espacial no contexto user/human - Context

**Gathered:** 2026-06-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Enriquecer **apenas a serialização** do contexto enviado ao LLM (`serializeContext` em
`src/llm/prompts.ts`) com **posição e distância** de blocos, mobs/entidades e jogadores,
reaproveitando dados que o `WorldSnapshot` **já captura** e hoje são descartados:
`nearbyBlockTypes[].examples` (posições), `EntityInfo.position` e `PlayerInfo.position`.

Hoje a serialização manda só `nome×contagem` dos blocos e só distância (sem posição) para
mobs/entidades/jogadores. Sem noção espacial o LLM decide às cegas (manda coletar um tronco
sem saber se está a 2m ou no topo de uma árvore inalcançável) — causa-raiz do "bot parado".

**Dentro do escopo:** mudar o texto produzido por `serializeContext` (mensagem **user/human**,
contexto). **Fora do escopo:** alterar o `WorldSnapshot`/tipos/snapshot builder; tocar o caminho
**assistant**; computar alcançabilidade real (raycast/pathfinding pré-check).

</domain>

<decisions>
## Implementation Decisions

### Formato da coordenada
- **D-01:** Formato **híbrido** por linha: posição **absoluta** `(x,y,z)` do exemplo + **distância**
  em metros + **Δaltura** (Δy relativo ao bot) como número cru. Ex:
  `oak_log×4 próx(12,70,-5) 8m Δy+7`.
- **D-01a:** Mantém a coordenada absoluta porque é consistente com o `pos=` do bot já no prompt e
  com o `target "x,y,z"` que `navigate` aceita (o LLM pode navegar a um ponto exato). O Δy é o que
  torna a altura legível para o raciocínio "perto/longe, acima/abaixo".

### Sinal de alcançável
- **D-02:** **Dados crus apenas** — renderizar o Δy como número (ex: `Δy+7`) e deixar o LLM inferir
  se é alcançável. **NÃO** emitir veredito textual ("no alto"/"inalcançável"/"fora de alcance").
- **D-02a:** Alcançabilidade real (raycast / pré-check de pathfinding) fica **fora de escopo** —
  é lógica pesada de outra fase. Aqui só expomos o dado espacial que o LLM precisa para decidir.
- **D-02b (reconciliação):** o Δy de D-01 é dado cru (sinal+magnitude), coerente com "sem rótulo"
  de D-02 — a direção (`+`/`-`) é leitura do número, não um julgamento de alcance.

### Orçamento (quanto incluir)
- **D-03:** Manter rico — **2-3 exemplos de posição por tipo** de bloco — MAS com **teto global**
  de coordenadas para blocos (~15-20). Ao exceder o teto, **priorizar os tipos mais próximos/úteis**
  (descartar os exemplos excedentes dos tipos menos relevantes; manter ao menos o count).
- **D-03a:** `examples` já vêm ordenados por distância (o `bot.findBlocks` ordena; o builder guarda
  os 3 primeiros encontrados — `src/perception/snapshot.ts`), então `examples[0]` é o mais próximo —
  selecionar os exemplos a renderizar é custo ~zero (sem reordenar).
- **D-03b:** Justificativa do teto: restrição carregada das fases anteriores — **D-07/D-17**: prompt
  longo degrada o modelo local e aproxima o timeout no caminho quente de decisão. O teto protege o
  Core Value (o loop não pode pendurar) sem abrir mão da riqueza espacial.

### Mobs / entidades / jogadores
- **D-04:** Incluir **posição** (além da distância já mostrada) para **todas as entidades já listadas**
  (~5) **e para jogadores**, no **mesmo formato híbrido** de D-01 (abs + distância + Δy).
- **D-04a:** Jogadores incluídos por consistência e porque a Phase 11 (modo assistente) se beneficia
  de saber onde o jogador está. `PlayerInfo.position` pode ser `null` (entidade fora do chunk) —
  degradar para "sem posição" sem quebrar (o serializer NUNCA lança).

### Claude's Discretion
- Sintaxe/pontuação exata das linhas (abreviações, separadores) desde que respeite D-01/D-03.
- Valor exato do teto global dentro da faixa ~15-20 (afinar conforme o tamanho real do prompt).
- Critério de "tipo mais útil" para priorização sob teto (ex: recursos prioritários do
  `buildDecisionGuide` — *_log > stone > minérios — vs. puramente o mais próximo).
- Como formatar `null`/ausência de posição (jogador fora do chunk).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `serializeContext()` (`src/llm/prompts.ts`): já monta o bloco user/human linha-a-linha de forma
  defensiva (tolerante a snapshot null e arrays vazios, NUNCA lança) — estender as linhas de blocos
  e entidades aqui, mantendo o mesmo estilo `lines.push(...)`.
- `WorldSnapshot.nearbyBlockTypes[].examples`, `entities[].position`, `players[].position`: dados
  espaciais já capturados — a fase é puramente de **renderização**, sem nova captura.
- `status.position` já no prompt (linha 137-139): base para calcular o Δy (Δaltura) por subtração.

### Established Patterns
- Slicing defensivo + limites compactos (`.slice(0, 8)` blocos, `.slice(0, 5)` entidades) já é o
  padrão de orçamento — o teto global de D-03 estende essa ideia para as coordenadas.
- `Math.round(...)` para distâncias/coords inteiras mantém o texto curto (já usado nas linhas atuais).
- `examples[0]` = mais próximo (findBlocks ordenado) — confiar nessa ordem, sem reordenar.

### Integration Points
- Nenhum novo. A saída de `serializeContext` já é injetada no caminho de AÇÃO/contexto user do loop
  cognitivo (`src/cognition/`); enriquecer o texto não muda assinaturas nem call-sites.
- Testes: `src/llm/prompts.test.ts` (se existir) deve cobrir o novo formato (presença de coord,
  respeito ao teto, tolerância a position null e snapshot null).

</code_context>

## Specific Ideas

- Exemplo-alvo de linha de bloco: `oak_log×4 próx(12,70,-5) 8m Δy+7` (count preservado + exemplo

  bem sozinho ao vivo.

  fora de escopo; candidata a fase própria (relaciona com Phase 10/12).
- **Memória espacial (POIs) no prompt** — já existe via Phase 08.1 (`POIs próximos:`); esta fase é
  complementar (percepção do raio atual), não mexe nos POIs.
