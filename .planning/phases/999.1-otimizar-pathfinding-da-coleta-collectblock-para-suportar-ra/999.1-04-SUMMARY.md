# Phase 999.1 Plan 04: Resolver double-wrap de executeWithSafety (D-06) Summary

Removido o double-wrap de `executeWithSafety`: o nó `execute` agora chama a skill diretamente, e cada skill registrada se auto-embrulha com seu próprio `progressChecker` (dig usa inventário, navigate usa `navigateTimeoutMs`), eliminando a camada externa com defaults genéricos que mascarava/duplicava o watchdog interno.

## What Was Built

### Task 1: Auditoria/auto-embrulho de navigate, follow, attack
- **navigate.ts**: já se auto-embrulhava corretamente em `executeWithSafety` usando `config.navigateTimeoutMs` + `progressChecker` de posição — nenhuma alteração necessária.
- **follow.ts** e **attack.ts**: são stubs que lançam imediatamente. Adicionado comentário explícito (D-06) documentando que stubs NÃO se auto-embrulham (não há operação async longa a proteger), com nota de que a implementação real futura deve envolver a operação contínua em `executeWithSafety`.
- **dig.ts**: já se auto-embrulha em dois caminhos (coleta por nome e mineração por posição) com `progressChecker` próprio — referência do padrão, sem alteração.

### Task 2: Remoção do wrap externo no nó execute (D-06)
- Trocado `await executeWithSafety(() => skillRegistry[skill!]!(bot, params))` por chamada direta `await skillRegistry[skill!]!(bot, params)`.
- Comentário atualizado para refletir que timeout/watchdog agora acontecem DENTRO de cada skill.
- Removido o import não-usado de `executeWithSafety` em nodes.ts.
- `try/catch`, `recordSuccess` e `recordFailure` preservados intactos.
