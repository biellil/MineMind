# Phase 999.1: Otimizar pathfinding da coleta (collectBlock) - Context

**Gathered:** 2026-06-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Permitir `PERCEPTION_RADIUS` alto (ex: 32) sem estourar memória/OOM no estado Gathering,
limitando o A* do `mineflayer-pathfinder` disparado por `bot.collectBlock.collect()`,
desacoplando o raio de busca de coleta do raio de percepção, e validando alcançabilidade
antes de coletar. A propriedade central da Fase 2 — **"o loop cognitivo nunca trava"** —
precisa permanecer verdadeira e ser ativamente verificada.

**Fora de escopo:** novos estados cognitivos, mudanças no LLM/motivação, alterar a
navegação (`navigate.ts` usa `findBlock` singular e NÃO contribui para o OOM de coleta).

</domain>

<root_cause>
## Causa-raiz (corrigida durante a discussão)

O todo de origem assumiu "bloqueio síncrono do event loop". A leitura do código de
`mineflayer-pathfinder` corrige essa premissa:

- O A* é **tick-sliced / event-driven**: computa ≤`tickTimeout` (40ms) por `physicsTick`
  e cede o event loop entre ticks. Não bloqueia o loop de forma síncrona.
- A causa real do "OOM sem timeout disparar": `searchRadius = -1` (default = **ilimitado**)
  desativa o gate `maxCost` do A* (`astar.js`), e `openHeap`/`openDataMap`/`closedDataSet`
  crescem **sem teto** quando o alvo é distante/inalcançável → a heap estoura (~78GB) e o
  processo é **morto pelo OOM killer ANTES** do timeout de 10s vencer.
- **Implicação:** o bound de `searchRadius` é **a correção da raiz**, não uma mitigação.

Fatos de API confirmados no código (importantes para o planner):
- `Movements.maxIterations` **NÃO existe** (premissa do todo era incorreta). Levers reais:
  `bot.pathfinder.searchRadius`, `bot.pathfinder.thinkTimeout` (5000ms), `tickTimeout` (40ms).
- `collectBlock.collect()` cria a **própria** instância de `Movements` (`new Movements(bot)`)
  e usa os globais `bot.pathfinder.*`. Bounds devem ser aplicados em **ambos**:
  `bot.pathfinder.*` E `bot.collectBlock.movements`, APÓS `loadPlugin(collectBlock)`.
- `collectBlock` pode **travar sem lançar erro** em obstrução inquebrável (issue #222).
- `bot.pathfinder.getPathTo(movements, goal, timeout)` é **síncrono**, não move o bot, e
  retorna `status: 'success' | 'partial' | 'noPath' | 'timeout'` — viável para pré-check barato.

</root_cause>

<decisions>
## Implementation Decisions

### Raio de busca (Área 1)
- **D-01:** Introduzir env **`GATHER_SEARCH_RADIUS` (default 16)**, independente de
  `PERCEPTION_RADIUS` (que volta/permanece 32). `dig.ts` passa a usar `GATHER_SEARCH_RADIUS`
  no `findBlocks({ maxDistance })`; percepção (`snapshot.ts`) continua usando `perceptionRadius`.
  Razão: ataca a fonte de candidatos distantes que alimentam o A*, mantendo percepção rica
  para o LLM; raio auto-documentado e tunável isolado.

### Bounds do A* (Área 2)
- **D-02:** Configurar **`searchRadius` ≈ 1.5–2× percepção (~48–64)** + **`thinkTimeout` ~2000ms**,
  expostos via env (ex: `PATHFINDER_SEARCH_RADIUS`, `PATHFINDER_THINK_TIMEOUT_MS`) com defaults
  sensatos, seguindo o padrão de `config.ts`. Limita a memória do A* na fonte (gate `maxCost`)
  + rede de segurança temporal secundária.
- **D-03:** Aplicar os bounds em **dois lugares** após `loadPlugin(collectBlock)` em
  `connection.ts`: nos globais `bot.pathfinder.thinkTimeout`/`searchRadius` E na instância
  `bot.collectBlock.movements` (que o collectBlock recria internamente).

### Alvo inalcançável (Área 3)
- **D-04:** Abordagem **híbrida**: rodar `bot.pathfinder.getPathTo(movements, GoalGetToBlock, timeoutCurto)`
  ANTES de `collect()` para descartar alvos inalcançáveis sem mover o bot nem disparar o hang #222.
  O resultado **alimenta a maquinaria de segurança existente** (`recordFailure`/cooldown/backoff de
  `safety.ts`) com rastro em memória — não inventar caminho paralelo de "pular silencioso".
- **D-05:** **Granularidade:** o pré-check escolhe uma **instância alcançável** entre os candidatos
  do tipo; o **cooldown por-tipo existente** (chave = nome do bloco, 15s) só dispara quando
  **NENHUMA instância do tipo é alcançável**. Mantém `safety.ts` sem mudar a chave do cooldown.

### Garantia "loop nunca trava" + verificação (Área 4)
- **D-06:** Resolver o double-wrap **removendo o wrap externo em `nodes.ts:207`**. A rede de
  segurança vive **dentro de cada skill** (onde mora o `progressChecker` específico — `dig.ts`
  já se auto-embrulha com checker de inventário). Regra: toda skill registrada se auto-embrulha
  em `executeWithSafety`.
- **D-07:** **Critério de aceitação** — smoke headless multi-tick (consistente com a verificação
  da Fase 2), com `PERCEPTION_RADIUS=32`, provando TRÊS coisas simultaneamente:
  1. **Sem OOM** — gathering de alvo inalcançável completa sem matar o processo (heap estável
     abaixo de um teto via `process.memoryUsage().heapUsed`).
  2. **Timeout/noPath dispara** — a skill rejeita (`SkillTimeoutError`/`SkillStuckError` ou
     `noPath`/`timeout` do pathfinder) dentro de `digTimeoutMs`.
  3. **Event loop responde** — heartbeat (`setInterval` ~50ms) mede lag e acusa **lag < 200ms**
     durante toda a execução da skill candidata a bloqueio. Este é o gate principal da propriedade.

### Claude's Discretion
- Nomes exatos das env vars novas e validação de range (seguir o padrão de `config.ts`).
- Valor exato do `timeoutCurto` do pré-check `getPathTo` (poucos ms/iterações).
- Forma do heartbeat util e do stub de pathfinder/mundo no smoke test (stub de alvo-inalcançável
  vs mundo headless real) — desde que prove os 3 pontos de D-07.
- Defaults finais dentro das faixas recomendadas (GATHER_SEARCH_RADIUS≈16; searchRadius≈48–64;
  thinkTimeout≈2000ms; lag≈200ms).

### Folded Todos
- **gathering-collectblock-oom** (`.planning/todos/pending/gathering-collectblock-oom.md`):
  é a origem desta fase. Sintoma (OOM ~78GB VM no Gathering), evidência e workaround atual
  (`PERCEPTION_RADIUS=8`) estão totalmente cobertos pelas decisões acima. Após o fix, restaurar
  o default de `PERCEPTION_RADIUS=32`.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **`safety.ts`** (anti-repeat N=3, cooldown por-tipo 15s, backoff→Idle): reusar para o resultado
  do pré-check (D-04/D-05) — não inventar maquinaria nova.
- **`executeWithSafety`** (`executor.ts`): timeout + watchdog de progresso via Promise.race; o
  heartbeat de D-07 pode reusar o padrão de `setInterval`.
- **`getPathTo`** (pathfinder): síncrono, não move o bot — base do pré-check de alcançabilidade.

### Established Patterns
- Env vars em `config.ts` com `|| default` + validação de range em startup (seguir para os novos raios/timeouts).
- Skills se auto-embrulham em `executeWithSafety` com `progressChecker` próprio (padronizar via D-06).

### Integration Points
- `connection.ts` `bot.once('spawn')`: único lugar onde `loadPlugin(collectBlock)` + `Movements` rodam
  — aplicar bounds aqui, após o load.
- Nó `execute` em `nodes.ts`: orquestra seleção de alvo (`highestPriorityGatherTarget`) e chamada da skill.
